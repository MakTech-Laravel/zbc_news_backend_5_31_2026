# robots.txt & XML sitemaps

Backend-served SEO submission assets, built with `spatie/laravel-sitemap`.

## Endpoints (routes/web.php)

| URL | Controller | Content |
|---|---|---|
| `/robots.txt` | `SitemapController@robots` | Allow public tree; disallow auth/admin/user/utility paths; `Sitemap:` directives |
| `/sitemap.xml` | `SitemapController@general` | Every public, indexable URL |
| `/news-sitemap.xml` | `SitemapController@news` | Google News sitemap, last 48h only |

### Why backend-generated (not static files)
- Sitemaps are inherently dynamic (DB-driven) — they must be code/routes.
- `robots.txt` is generated by the same controller so its `Sitemap:` URLs and the
  disallow list live in one versioned place and use `config('app.frontend_url')`
  (env-aware). Laravel's default `public/robots.txt` was **removed** because a static
  file in `public/` shadows the route.

### Reverse-proxy requirement (production)
The public origin is the **SSR frontend** (Node / `react-router-serve`). The reverse
proxy (nginx) must route `/robots.txt`, `/sitemap.xml`, `/news-sitemap.xml` (and the
existing `/share/`) to the **backend**, everything else to the frontend. That keeps the
sitemaps on the main content domain (best for SEO) while the backend generates them.
The `Sitemap:` URLs and every `<loc>` use `config('app.frontend_url')`, so **that must be
set to the real origin** (see the SSR plan's pre-deploy checklist) or the URLs are wrong.

## What counts as a public URL (no divergence)

`SitemapService` derives URLs from the **same published/active definitions the SEO
resolver uses**, so the sitemap can't drift from what actually resolves:

- Home `/`
- Static pages: non-template `seo_pages` rows (excluding `home`, `news-details`, and
  `category-*` per-category rows) → about, contact, privacy, terms, cookie-policy,
  accessibility-statement, advertise, careers, newsletter
- Categories: `status = active`
- Articles: `status = published`
- Authors: users with ≥1 published article and a slug
- Tags: tags used by ≥1 published article, slugified exactly as the frontend `slugifyTag()`

`<lastmod>` = each entity's `updated_at`.

### noindex is respected (excluded, not just tagged)
A URL is omitted when its **governing `seo_pages` row** has `noindex` — the same
governing-row rule as `SeoResolverService::robotsFor` (template row, or a per-category
exact row that overrides it). A test asserts a page excluded from the sitemap also
resolves to `noindex,nofollow`, guarding against the two rules diverging.

## News sitemap (Google News)

- Only articles with `published_at` within **48 hours**; older ones drop out entirely.
- `<news:publication_date>` is byte-identical to the article's JSON-LD `datePublished`
  from `SeoResolverService` (both Carbon `toW3cString()` / `toIso8601String()` — verified
  equal, and asserted by a test that compares the sitemap entry to the resolve endpoint).
- `<news:name>` / `<news:language>` come from site settings.
- Element order (`publication` → `publication_date` → `title`) is fixed via an overridden
  `resources/views/vendor/sitemap/news.blade.php`, because spatie ships `title` before
  `publication_date`, which fails strict Google News XSD validation.
- Capped at **1000 URLs** (Google's per-file max), newest first. If article volume ever
  exceeds 1000 published in 48h, split into a sitemap index — **follow-up**, flagged here;
  realistic volume is far below this.

## Caching strategy

On-demand with `Cache::remember`:
- general: **1 hour** TTL (content URLs change slowly)
- news: **10 minutes** TTL (the 48h window shifts constantly)

Both rebuild lazily on the next request after expiry — never per-request from scratch.
`php artisan sitemap:refresh` flushes + warms both (call it after bulk publish/SEO edits);
it's scheduled hourly to keep the general cache warm.

## Validation

`tests/Feature/Seo/SitemapTest.php` (fetches the official XSDs, doesn't assume):
- general validates against `sitemap.org/schemas/sitemap/0.9/sitemap.xsd`
- news validates against Google's `sitemap-news/0.9/sitemap-news.xsd` (imported into a
  sitemaps-namespace `urlset` schema)
- includes a known article; excludes a `noindex` page; news within 48h only;
  `news:publication_date` == resolve JSON-LD `datePublished`.
